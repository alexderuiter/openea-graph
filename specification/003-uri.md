# URI identity

## Requirements

Resource URIs MUST be globally unique within their intended graph context and
SHOULD remain stable over time, including when display names change.

## Recommended schemes

Repositories MAY use resolvable HTTPS URIs, URNs or another valid absolute URI
scheme. A URI does not have to resolve to a public website.

Examples:

```text
oea://municipality.example/application/djuma
https://architecture.example.nl/resource/application/djuma
archimate://element-type/application-component
urn:openea:workspace:alex:amsterdam:component:zaaksysteem
```

## Slugs

Human-readable slugs are recommended, but a URI MUST NOT be treated as a mutable display name.

## Namespaces

A repository SHOULD declare namespace prefixes in `metadata.json` for compact
identifiers, display and export. When an identifier begins with a declared
`<prefix>:`, implementations expand it by replacing that prefix and colon with
the declared namespace value before URI validation and identity comparison.

For example:

```yaml
namespaces:
  ams-work: urn:openea:workspace:alex:amsterdam:
  openea: https://openea.dev/ns/
  gemma: https://gemmaonline.nl/id/
```

The compact identifier `ams-work:component/zaaksysteem` expands to
`urn:openea:workspace:alex:amsterdam:component/zaaksysteem`.

## Namespace governance and provisional identifiers

Workspace and provisional namespaces are valid. The recommended pattern is:

```text
urn:openea:workspace:<owner-or-project>:
```

The owner or project segment is a local disambiguator and does not need to
correspond to an official organization. A workspace namespace MUST NOT imply
that it is an official namespace of an organization. For example, a workspace
segment named `amsterdam` can describe the subject or scope of working material;
it does not make that namespace an official Gemeente Amsterdam namespace.

Resources can later be mapped to canonical identifiers through explicit mapping
relationships or migrated through reviewable changes. Early modeling work
remains valid when official identifiers become available: its original
workspace URIs continue to identify the resources and preserve repository
history unless an explicit migration changes them.

Workspace URIs are ordinary URIs. OpenEA Graph does not assign them a mandatory
identifier status and does not require a governance object or profile.
